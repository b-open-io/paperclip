EXECUTIVE BRIEF: AIP SIGNATURE → BAP IDENTITY FLOW
═══════════════════════════════════════════════════════════════════════════════

RESEARCH OBJECTIVE
─────────────────
Determine the exact path from AIP (Author Identity Protocol) signer address to
verified BAP (Bitcoin Attestation Protocol) identity in ClawNet skill publishing.

FINDING: NO DIRECT PATH EXISTS
─────────────────────────────

The system uses TWO INDEPENDENT IDENTITY STREAMS:

  STREAM 1: AIP SIGNATURE
  ├─ Private Key (signing key from CLI config)
  ├─ Process: AIP.sign(skillData)
  ├─ Output: signerAddress (Bitcoin address)
  └─ Current Use: Stored in database (not used for identity)

  STREAM 2: USER AUTHENTICATION ← PRIMARY IDENTITY SOURCE
  ├─ Identity Key (from BRC-100 wallet or Sigma Auth)
  ├─ Process: BRC-31 header validation OR Sigma token verification
  ├─ Output: authorBapId (via pubkeyToBapId or Sigma session)
  └─ Current Use: Author identification in database

These use DIFFERENT keys for DIFFERENT purposes.


KEY INSIGHT
──────────

In /Users/satchmo/code/clawnet/app/api/v1/skills/route.ts (line 91):

  const result = await getConvex().mutation(api.mutations.publishVersion, {
    authorBapId: auth.bapId,  ← FROM AUTHENTICATION LAYER
    aipSignature: payload.aipSignature,  ← FROM REQUEST (stored only)
    signerAddress: payload.signerAddress,  ← FROM REQUEST (stored only)
  });

The server IGNORES signerAddress from the request and uses auth.bapId instead.

This is by design: separates signing proof from identity claim.


AUTHENTICATION SOURCES
──────────────────────

Option 1: BRC-31 (Preferred)
├─ Extract: x-bsv-auth-identity-key header (66 hex chars)
├─ Validate: ECDSA signature against BRC-43 derived keys
├─ Derive: pubkeyToBapId(identityKey) → BAP ID
└─ Result: { bapId, pubkey }

Option 2: Sigma Bearer Token
├─ Extract: Authorization: Bearer <token>
├─ Validate: Call auth.sigmaidentity.com/api/auth/get-session
├─ Derive: session.activeOrganizationId || user.pubkey
└─ Result: { bapId, pubkey }

PROFILE RESOLUTION
──────────────────

When rendering skill detail page:

  1. Load skillVersions.authorBapId
  2. Query Convex profiles table (cache)
     └─ Cache HIT? Use stored profile
     └─ Cache MISS? Fetch from Sigma API
  3. Call: GET https://sigma.1sat.app/1sat/bap/profile/{bapId}
  4. Response: { alternateName, image, description, ... }
  5. Render: Author name, avatar, bio


API CONTRACTS
─────────────

POST /api/v1/skills
├─ Auth: BRC-31 headers OR Bearer token
├─ Payload: { slug, name, files, aipSignature, signerAddress, ... }
└─ Result: authorBapId = auth.bapId (from headers/token, not payload)

GET https://sigma.1sat.app/1sat/bap/profile/{bapId}
├─ Input: bapId (path parameter)
├─ Success: { status: "OK", result: SigmaProfile }
└─ Not Found: { message: "Profile not found..." }


COMMON MISTAKES TO AVOID
────────────────────────

❌ Don't assume signerAddress = authorBapId
   They come from different keys. No relationship expected.

❌ Don't verify signerAddress against auth.pubkey
   The signing key and identity key are intentionally separate.

❌ Don't block publishing if BAP profile doesn't exist
   Profiles are optional. User can create one later.

❌ Don't use signerAddress for author identification
   Always use authorBapId (from authentication layer).

❌ Don't query Sigma API without caching
   Profile data doesn't change frequently. Cache locally.

BOTTOM LINE
───────────

To go from AIP signerAddress → BAP identity:

  1. AIP produces signerAddress ✓ (stored for audit)
  2. Authentication layer produces authorBapId ✓ (primary identity)
  3. authorBapId → Sigma API → profile { name, avatar, bio } ✓
  4. Render skill with author info ✓

NO DIRECT TRANSFORMATION from signerAddress to authorBapId.
The authentication layer is the sole source of truth.

This design allows:
  • Different keys for different security domains
  • Future delegated publishing (sign with one key, publish with another)
  • Clear audit trail (who signed vs. who published)
  • Flexible identity management (orgs, team members, etc.)
